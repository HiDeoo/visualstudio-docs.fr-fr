# <a name="c26115"></a>C26115
AVERTISSEMENT C26115 : échec de la libération du verrou \<lock > dans la fonction \<func >.

 La mise en œuvre de paires d' *acquisitions* de verrous d’étendue syntaxiqueC++ et de verrous de libération dans les programmes C/n’est pas effectuée par le langage. Une fonction peut introduire un effet secondaire de verrouillage en effectuant une modification observable de l’état de concurrence. Par exemple, une fonction de wrapper de verrou incrémente le nombre d’acquisitions de verrous, ou nombre de verrous, pour un verrou donné.

 Vous pouvez annoter une fonction qui a un effet secondaire à partir d’une libération d’acquisition ou de verrouillage de verrou à l’aide de `_Acquires_lock_` ou `_Releases_lock_`, respectivement. Sans ces annotations, une fonction est supposée ne pas modifier le nombre de verrous après qu’elle a été retournée. Si les acquisitions et les mises en production ne sont pas équilibrées, elles sont considérées comme *orphelines*. L’avertissement C26115 est émis lorsqu’une fonction introduit un verrou orphelin.

## <a name="example"></a>Exemple
 L’exemple suivant génère un avertissement C26115, car il existe un verrou orphelin dans une fonction qui n’est pas annotée avec `_Acquires_lock_`.

```

typedef struct _DATA
{
    CRITICAL_SECTION cs;
} DATA;

void FailToReleaseLock(int flag, DATA* p)
{
    EnterCriticalSection(&p->cs);

    if (flag)
        return; // Warning C26115

    LeaveCriticalSection(&p->cs);
}
```
